バーコードの核となる物理学: ハードウェアがピクセルとラインをデコードする方法

バーコードは基本的に、記号の幅と間隔 (またはモジュール値) をデータにマッピングする機械可読エンコード システムです。線形 (1D) バーコードは、正確な幅比を持つバーとスペースのシーケンスとして情報を表します。レーザー スキャナは、コード上でレーザー ビームを走査しながら反射光の強度を読み取ります。スキャナはアナログ反射率曲線をデジタル波形に変換し、要素の幅を測定し、エンコードされた値を再構築します。要素の幅、コントラスト、またはクワイエット ゾーンの小さな偏差により、ビット境界が反転し、誤読が発生する可能性があります。

2D マトリックス コード (データ マトリックス、QR、PDF417) は、エリア イメージャー (基本的には画像マトリックスをサンプリングするカメラ センサー) によってキャプチャされます。イメージャは、幾何学的補正 (デスキュー)、二値化、およびファインダー パターン検出を実行して、タイミング セルと方向を特定します。リードソロモン (通常はデータ マトリックスの ECC 200) などのエラー訂正コードにより、部分的な損傷に対して高い生存性が可能になります。デコーダはパリティ シンボルから欠落したデータを再構築します。

操作上の注意: バーコード スキャナは、任意の視覚的変更を許容しません。バーコードは、寸法、クワイエット ゾーン、コントラスト、およびチェック ディジットがハードウェアが期待する「文法」である制約された印刷システムです。印刷時のわずか 0.01 mm のずれでも、高速チェックアウトや自動仕分けラインでは問題が発生する可能性があります。

1. コード 128 — 高密度物流のための企業標準

Code 128 は、輸送、物流、および英数字ペイロードを必要とするアプリケーションで広く使用されているコンパクトで高密度の線形シンボルです。その柔軟性は、A、B、C の 3 つのコード セットによってもたらされます。各コード セットは、特定の文字と密度に合わせて最適化されます。コード C は、数値データをペア (コード シンボルごとに 2 桁) としてエンコードし、数値密度を 2 倍にします。

構造とエンコーディング

一般的な Code 128 シンボルには、スタート パターン (スタート A/B/C)、一連のデータ シンボル (それぞれコード値 0 ～ 106 を表す)、チェック デジット (モジュロ-103)、およびストップ パターンが含まれます。各シンボルは、3 つのバーと 3 つのスペースに分割された 11 個のモジュールで表されます。物理的な幅は、モジュール幅 (X 次元) とシンボルの数によって決まります。

コードセット

  • コードセットA: 大文字、制御コード、特殊文字。
  • コードセットB: 大文字と小文字、および標準の印刷可能な ASCII。
  • コードセットC： 数値専用圧縮モード: 数字のペアを 1 つのシンボルとしてエンコードします (数値文字列の場合は高密度)。

Modulo-103 チェック デジット

チェック ディジットの計算には、スタート キャラクタと各データ シンボルの数値が使用されます。合計を計算します: (StartValue) + (1 * valueOfSymbol1) + (2 * valueOfSymbol2) + ... + (n * valueOfSymbolN)。チェック ディジットは、この合計を 103 で割った余りです。

// Example: pseudo-calculation
StartValue = 104 // (example for Start B)
values = [50, 45, 12] // symbol values
sum = StartValue + 1*50 + 2*45 + 3*12
check = sum % 103

密度の最適化

長い数値シーケンス (日付、GTIN フラグメント) をエンコードする場合は、数値ペアについてはコード C に切り替えることを推奨します。スマート エンコーダはペイロードを分析し、コード セットを切り替えてコンパクトさを最大限に高める必要があります。多くのジェネレーター実装は、コード セットの自動最適化を提供します。ライブラリが文字列の途中での切り替えを処理し、セットの変更全体でチェック ディジットが適切に再計算されていることを確認してください。

実装に関する考慮事項

  • スキャナの公差 (一般的な値: 0.19 ～ 0.33 mm) を満たす X 寸法を選択します。
  • 両側に少なくとも 10 × X のクワイエット ゾーンを確保します。
  • 可能な場合は、スケールに依存しない出力のためにバーコードをベクター SVG としてレンダリングします。

ビジュアル アセット プレースホルダー: 開始、データ、チェック、および停止パターンを示す注釈付きコード 128 の内訳。

2. UPC-A および EAN-13 — 世界的な小売流通の基盤

UPC-A と EAN-13 は、販売時点で使用される正規の小売シンボルです。どちらも数値のみで、チェックアウトを高速にスキャンできるように設計されています。 UPC-A は 12 桁 (チェック ディジットを含む) をエンコードし、北米で普及しています。 EAN-13 は 13 桁をエンコードし、国際標準です。左端の数字は通常、国または番号付けシステムの識別子として機能します。

UPC-A の構造

レイアウト：[数値体系桁(1)][メーカーコード(5)][製品コード(5)][チェックデジット(1)]。チェック ディジットは、奇数桁の合計 + 3 × 偶数桁の合計という、Modulo-10 加重スキームを使用します。 10 の倍数になるようにこの合計に加算する最小の数値がチェック ディジットです。

EAN-13 の相違点

EAN-13 は 13 桁目を追加し、UPC-A と同様のガード パターンを使用します。小売店で展示される場合、EAN-13 と UPC-A は多くの POS システムで機能的に相互運用可能ですが、エンコードが正しい必要があります。よくあるエラーは、数字以外の文字または先頭のゼロを誤ってエンコードしようとすることです。これらにより、デコーダはコードを拒否します。

実践的なルール

  • 常に正しい桁数を指定してください。 GTIN ルールを完全に理解していない限り、プログラムでパディングや切り詰めを行わないでください。
  • シンボルの両側に少なくとも 9 × X のクワイエット ゾーンを維持します。
  • バーコードの高さ (ガード バーを含む) が POS スキャナーの要件を満たしていることを確認します。公称 X 寸法での UPC-A の一般的な最小高さは約 22.85 mm ですが、検証済みの印刷およびスキャン テストにより、限られたスペースではこれより低い高さが使用されます。

3. データ マトリックスと QR コード — 高密度 2D 産業用アレイ

データ マトリックスと QR コードは、暗/明モジュールのグリッドにデータを保存するマトリックス 2D シンボルです。これらは、ファインダー/タイミング パターン、ECC、および一般的なペイロード サイズが異なります。データ マトリックス (ECC 200) は、リードソロモン誤り訂正による回復力を維持しながら、非常に小さな印刷サイズをサポートするため、航空宇宙、医療、電子機器で広く使用されています。 QR コードは消費者のユースケースで広く普及しており、ブランド化のためのより大きな目に見えるロゴとより高いエラー修正レベルをサポートしています。

データマトリックス (ECC 200)

データ マトリックスは、ソリッド L 字型ファインダー (「ファインダー パターン」) と反対側のタイミング パターンを使用します。 ECC 200 はリードソロモン コードを使用し、最大サイズの最大 2335 文字の英数字、またはシリアル化された識別子に適した網羅的なバイナリ ペイロードをサポートします。 Data Matrix は、小さな表面 (特殊な印刷では最大 2 ～ 3 mm のモジュール サイズ) およびダイレクト パーツ マーキング (DPM) に優れています。

QRコードの仕組み

QR codes use three square finder patterns in corners and an alignment/timing grid. Error correction levels (L, M, Q, H) let you trade capacity for resiliency; H (30% recovery) allows larger logos but reduces raw data capacity. QR's broad adoption makes it ideal for links, vCards, and consumer media.

ダメージとエラー訂正の比較

激しい摩耗テストでは、ECC 200 を備えたデータ マトリックスは、非常に小さなマーキング領域で QR よりも優れたパフォーマンスを発揮することがよくあります。これは、ECC ブロックの分布が異なっており、一部の産業用イメージャーではデータ マトリックスのファインダー パターンにより、より正確な幾何学補正が容易になるためです。

比較グラフのプレースホルダー: 表面積別の推奨シンボル、予想される最大回復可能ダメージ率、一般的なエンコーダ ライブラリ設定。

4. コード 39 — 自動車および軍事物流におけるレガシーの信頼性

Code 39 は、43 文字 (0 ～ 9、A ～ Z、および 7 つの特殊文字) をエンコードする古い個別のシンボルです。各文字は、3/9 ワイド バー ルールに従って、5 つのバーと 4 つのスペース (9 要素) で表されます。各文字の幅が比較的広いため、Code 39 はスペース効率が良くありませんが、レガシー システム、産業用途、およびチェック ディジットを必要とせずに単純なエンコードが必要な場合には依然として人気があります。

フル ASCII と拡張子

コード 39 は、完全な 128 ASCII セットを表すペアを使用してフル ASCII モードで拡張できますが、これにより幅がさらに増加し​​ます。大きなテキスト ペイロードの場合は、Code 128 または 2D マトリックスを推奨します。

コード 39 が生き残る理由

そのシンプルさ、堅牢なハードウェア サポート、低品質印刷への耐性により、Code 39 は一部の自動車および軍事サプライ チェーンで使用されています。ただし、より広い印刷フットプリントを想定し、狭い要素 (N) と広い要素 (W) の比率に対するスキャナの許容誤差を確認してください。

5. ITF-14 (Interleaved 2 of 5) — 段ボール包装の最適化

ITF-14 は、カートンや外装パッケージにマーキングするために設計されています。これは、数字をペアでエンコードするインターリーブ数値シンボルです。1 つの数字はバーに寄与し、ペアの数字はスペースに寄与します。このインターリーブにより、数値エンコードがコンパクトになり、単純な数値文字列 (GTIN-14 など) が読みやすくなります。 ITF-14 シンボルにはベアラー バー (コードの上下にある太いバー) が含まれることが多く、スキャナーがコード全体を確実に読み取ることで部分的なスキャンを防止し、カートンの端の凹凸によって引き起こされるショート スキャンを防ぎます。

印刷に関する考慮事項

  • 段ボールに印刷する場合は、コンベア ラインで先頭/末尾が切り取られるリスクを軽減するために、厚いベアラー バーを使用します。
  • フレキソ印刷またはダイレクトサーマル印刷の許容誤差を検証します。カートンの表面は粗く、インクまたは熱転写の動作が問題になります。
  • 最小 X 寸法は、表面の粗さのため、通常、小売コードよりも大きくなります。

よくある質問 (FAQ)

UPC-A で小文字をエンコードできないのはなぜですか?
UPC-A は、POS システムでの相互運用性を考慮した設計により、厳密に数値化されています。数値以外のデータをエンコードするとシンボル規則に違反し、小売店のスキャナでは読み取れない出力が生成されます。
クワイエット ゾーンとは何ですか?また、その広さはどれくらいでなければなりませんか?
クワイエット ゾーンは、デコーダーがシンボル境界を検出するために必要な、バーコードの周囲の印刷されないマージンです。線形コードの場合、一般的な最小値は各辺で 10 × X (モジュール幅) です。 EAN/UPC の場合、多くの場合 9 × X が使用されます。 2D コードの場合、クワイエット ゾーンの推奨事項は異なりますが、通常は仕様に応じてシンボルの周囲に 1 ～ 2 つのモジュールです。
標準のレーザー スキャナはデータ マトリックスをデコードできますか?
いいえ — Data Matrix にはエリア イメージャ (カメラ スタイルのスキャナ) が必要です。レーザー ライン スキャナは線形反射率パターンのみを読み取り、2D マトリックス グリッドを再構成できません。
チェックデジットが間違っている場合はどうなりますか?
デコーダは通常、シンボルを無効として拒否するか、チェックサム エラーを生成します。チェック ディジットは、カットまたは汚れたシンボルの誤解釈を防ぐ一次整合性チェックです。
Code 39 のバーコードが非常に長く見えるのはなぜですか?
Code 39 は、すべての文字を比較的広いパターンとしてエンコードします。長い英数字文字列の場合、スペース効率が悪くなります。長いペイロードの場合は、Code 128 または 2D シンボルを検討してください。
オープンソースのシンボルと独自のシンボルの違いは何ですか?
オープン スタンダード (例: GS1 マネージド エンコーディング) では仕様が公開されており、広範囲にわたるスキャナーがサポートされています。独自のシンボルは機能を提供する可能性がありますが、ベンダーロックインのリスクがあります。相互運用性のための標準を好みます。
1D ではなく 2D を選択する必要があるのはどのような場合ですか?
ペイロード サイズが快適な線形幅を超える場合、バイナリまたは圧縮データが必要な場合、またはマーキング スペースが限られている場合は、2D を選択してください。 POS での単純な数値の製品コードの場合は、互換性を確保するために 1D を使用してください。
GS1 アプリケーション識別子 (AI) コードを生成するにはどうすればよいですか?
GS1 対応エンコーダを使用して、アプリケーション識別子を人間が読めるフィールドの括弧内にフォーマットし、AI メタ構造をバーコード ペイロードにエンコードします (GTIN の場合は (01))。 GS1 フォーマット規則に従って、正しい AI とフィールド長を確保してください。

概要: シンボル アーキテクチャの選択

このクイック デシジョン ツリーを使用します。

  • 小売製品のバーコード → UPC-A / EAN-13
  • 産業物流と長い英数字 ID → コード128
  • 極小表面または部品への直接マーキング → データマトリックス
  • 消費者リンク/マーケティング → QRコード

最終アクション ループ: 以下のジェネレータ パネルを使用して、一般的な出荷 X 寸法に最適化された生産対応の Code 128 プリセットを構成するか、小型部品用のデータ マトリックス プリセットに切り替えます。実稼働用にベクター SVG をエクスポートし、印刷実行パイプライン用に検証済みのペイロードを安全な CSV に保存します。
