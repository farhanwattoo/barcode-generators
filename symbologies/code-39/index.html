コード 39: ワイド文字、フル ASCII 拡張子、およびレガシー ロジスティックス

Code 39 が生き残っているのは、シンプルでよく理解されており、従来の産業環境で広くサポートされているためです。欠点は幅です。すべての文字は、Code 128 などの密度の高いシンボルよりも多くの水平スペースを消費するため、長いペイロードは小さなラベルではすぐに扱いにくくなります。

基本シンボルでは、大文字、数字、および少数の句読点がサポートされています。完全な ASCII 拡張子は存在しますが、一部の文字は複数の文字の組み合わせで表現されるため、印刷される長さはさらに長くなります。

強さ制約
シンプルなレガシーサポート設置面積が大きい
古いハードウェアでも読み取り可能限定されたネイティブ文字セット

コード 39: ワイド文字、フル ASCII 拡張、およびレガシー ロジスティクスは、ペイロードに必要な文字セット、利用可能な水平方向または正方形のスペースの量、すでに導入されているスキャナ、およびシンボルが小売業者、GS1、または業界固有の標準を満たす必要があるかどうかという 4 つの運用上の質問にマッピングすると、最も簡単に評価できます。この決定フレームにより、より適切な技術的適合性が存在する場合に、チームが使い慣れたコード タイプを選択することがなくなります。

  • ペイロード ルールから始めます。数値のみ、混合テキスト、バイナリ データ、または構造化識別子は、それぞれ異なるシンボルを指します。
  • クワイエット ゾーン、倍率、スキャナーの距離が選択プロセスの一部となるように、物理的な表面を早めにチェックしてください。
  • ユースケースが小売 POS、社内物流、部品の直接マーキング、または顧客対応のモバイル スキャンであるかどうかを確認します。
  • ジオメトリやコントラストが不十分な場合、技術的に正しいシンボルでも失敗する可能性があるため、エンコーダ検証と印刷検証を組み合わせてください。

検証に関する一般的な質問

Code 39 ガイドでは、その古典的な文字セット、フル ASCII 拡張機能、およびそれが依然として従来のサプライ チェーンに存在する理由について説明します。プロダクションアートワークを承認する前に、正確な文字数制限、チェックサムまたはエラー修正の動作、スキャナーの互換性、および人間が判読できる書式設定ルールを確認してください。これらのチェックにより、チームは、選択したバーコードがソフトウェアでは機能するが、スキャンする必要がある実際のチャネルでは機能しないことをプロセスの後半で発見することを防ぎます。

FAQ

コード 39 がコード 128 よりも長く見えるのはなぜですか?

Code 39 は密度の低いシンボルであり、各文字をより幅広いパターンでエンコードするためです。
